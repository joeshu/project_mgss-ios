# MGSS 下一轮 UI / 图形视觉方向 Brief 与验收标准

> 角色：product-director  
> Run：`legion-20260429_063338-project-mgss-ios-optimize`  
> 适用范围：`Sources/ProjectMGSS/GameView.swift`、`Sources/ProjectMGSS/GameScene.swift` 及后续原创图形资产接入  
> 重点阶段：选房阶段、夜间防守阶段、HUD / 色彩 / 图形资产  
> 版权约束：只做原创视觉表达，不复制、临摹或复刻任何商业原版素材、角色、UI、图标、动效或版式。

## 1. 产品目标

下一轮视觉优化的目标不是“堆美术”，而是把当前可玩原型升级为**信息清晰、阶段辨识强、低成本可落地的移动端塔防宿舍体验**。

核心原则：

1. **阶段一眼可辨**：选房阶段偏“规划 / 安全屋选择”，夜间防守偏“压迫 / 资源决策 / 危机响应”。
2. **玩家不迷路**：地图只承载位置、风险、门、床、炮台槽位、敌人动线；关键操作仍由 HUD 和底部操作区完成。
3. **小屏优先**：iPhone 小屏上优先保证文字、按钮、血条、资源数字可读；图形资产不能挤占核心交互区域。
4. **原创轻量资产**：图形风格应服务 SpriteKit + SwiftUI 的快速集成，优先使用可切片、可复用、透明背景 PNG / WebP / PDF-like vector 风格资产。
5. **避开商业原版**：不得使用“猛鬼宿舍”或任何现有商业游戏的具体角色造型、UI 排版、按钮样式、地图纹理、房间布局视觉和图标符号。

## 2. 视觉方向总纲

### 2.1 关键词

- **校园夜巡**：深夜宿舍、走廊冷光、门缝警戒、简化俯视图。
- **可爱压迫感**：敌人有威胁但不血腥；偏卡通剪影、发光眼、雾气轮廓。
- **霓虹战术 HUD**：深色底 + 高对比边线 + 资源芯片 + 阶段色。
- **低噪声地图**：房间和走廊只做必要质感，避免细碎装饰影响可读性。

### 2.2 美术风格

建议方向：**原创 2.5D 扁平俯视 + 霓虹描边 + 柔和体积光**。

- 地图：俯视 / 轻等距皆可，但最终需兼容当前 `GameScene` 的 2D 坐标和节点布局。
- UI：SwiftUI 卡片式玻璃暗面板，边框按状态变色。
- 图标：圆角、厚描边、简单几何符号，适合 24–48 px 小尺寸识别。
- 角色：不强调复杂人物，玩家可为“睡衣学生 / 被窝小人 / 床上轮廓”；敌人为原创“夜巡幽影 / 噪声怪 / 影雾入侵者”。

## 3. 阶段一：选房阶段 Brief

### 3.1 当前状态理解

代码中选房阶段已有：

- 顶部：`选择宿舍`、候选房间数、当前房间。
- 底部：当前选择、风险、产出、门耐久、横向房间切换、确认入住按钮。
- 地图：4 个宿舍房间 + 中央走廊 + 已选房间高亮 + 炮台槽位提示。

### 3.2 体验目标

选房阶段要传达：**“这是入夜前唯一的战略选择，选完后角色固定，不再自由走动。”**

玩家在 3 秒内应理解：

1. 现在不是战斗，而是在选房。
2. 4 个房间差异来自风险、收益、门耐久和位置。
3. 地图用来看位置和路线，不是操控角色移动。
4. 选中房间将影响开局难度。

### 3.3 视觉表现要求

#### 地图

- 已选房间：暖黄色 / 琥珀色描边，内侧轻微暖光。
- 未选房间：低饱和蓝紫灰，保持可见但退后。
- 风险等级：可在房间角落用 1–4 个小点 / 小三角 / 竖条表示，不能只依赖文字。
- 收益倾向：可使用金币小符号或“+”视觉标记，但不得让地图变成商店面板。
- 门位置：必须清晰，门是选房风险判断关键。
- 走廊：中央冷色通道，暗示敌人将从走廊压迫，但选房阶段不应出现强烈警报红。

#### UI 面板

- 顶部说明应保持轻量，不超过两行。
- 底部“当前选择”卡片应成为视觉主焦点，其优先级高于地图装饰。
- “确认入住并开始夜晚”按钮应使用明确的开始态：绿色 / 青绿色，搭配月亮或门锁符号。
- 风险色建议：
  - 风险 1：绿色 `#6FE38B`
  - 风险 2：黄色 `#FFD766`
  - 风险 3：橙色 `#FF9E45`
  - 风险 4：红橙 `#FF5E57`

### 3.4 文案与信息层级

优先级从高到低：

1. 当前选择房名。
2. 风险 / 产出 / 门耐久三指标。
3. 确认入住按钮。
4. 候选房间切换。
5. “人物固定，不自由走动”的规则提醒。

验收时不接受：地图很漂亮但选房指标不突出。

## 4. 阶段二：夜间防守阶段 Brief

### 4.1 当前状态理解

代码中夜间防守阶段已有：

- 顶部 HUD：时间、威胁状态、金币、电力、门等级、炮台数、房门 / 敌人血条。
- 中央地图：玩家、床、门、敌人、炮台、掉落道具、射线、警报气泡。
- 底部操作：睡觉 / 醒来、商店、道具策略、一键修门。
- 危急 Overlay：冻结、破门、修门建议。

### 4.2 体验目标

夜间防守要传达：**“我被固定在房间内，核心循环是睡觉发育 → 醒来布防 / 升级 → 修门救急 → 撑到天亮或击退敌人。”**

玩家应能持续判断：

1. 还有多久天亮。
2. 门是否危险。
3. 敌人处于巡查、逼近、破门、狂暴哪种状态。
4. 当前最推荐做什么。
5. 哪些资源可立刻转化为防守能力。

### 4.3 视觉表现要求

#### 地图战斗态

- 敌人动线：从走廊向所选房间门口压迫，建议加入轻量路径暗纹或脚步 / 雾气残影。
- 破门态：门节点应有红橙闪烁边、裂纹、震动感或冲击波；但不得遮挡血条和按钮。
- 睡觉态：玩家 / 床应出现柔和蓝青睡眠光环，表达“正在产出资源”。
- 炮台：用几何炮台底座 + 发光炮口，等级差异可通过尺寸、描边层数、核心颜色表现。
- 冻结态：敌人变为青蓝，附冰晶 / 低透明霜层；冻结状态要明显不同于普通蓝色 UI。
- 道具掉落：小尺寸仍可识别，建议统一圆形徽章底 + 内部符号。

#### HUD 战斗态

- 顶部 HUD 是“战况总览”，不得被地图背景淹没。
- 时间“距天亮 Xs”必须高可读，建议黄色 / 白色，不随警报变红到不可读。
- 门血条优先级高于敌人血条；门血条低于 55% 后变橙，低于 25% 后变红并触发强提示。
- 推荐动作需保持单一，不同时高亮多个主操作按钮。
- 危急 Overlay 应只在强事件出现：破门、门血低、冻结窗口、修门建议；不能常驻刷屏。

### 4.4 情绪节奏色彩

按夜间时间推进：

1. **发育期 0–45s**：绿色 / 青色，低压，鼓励睡觉和升床。
2. **守门期 45–110s**：橙色，强调补炮台、升门、看门血。
3. **反击期 110–180s**：红色 / 品红，压迫感提升，但信息仍需清晰。
4. **冻结 / 控制窗口**：冰青色，短时覆盖敌人和提示，不改变资源系统颜色。
5. **胜利天亮**：暖金 / 淡蓝晨光，明确从夜色中解脱。
6. **失败失守**：低饱和红黑，不使用血腥画面。

## 5. HUD / 色彩 / 图形资产规范

### 5.1 色彩 Token 建议

| 用途 | 建议色 | 说明 |
|---|---:|---|
| 背景深夜 | `#070817` / `#130B22` | App 背景渐变 |
| 面板底 | `#11111C` 80–92% | SwiftUI HUD 卡片 |
| 走廊冷紫 | `#1B1630` | 地图主底色 |
| 房间未选 | `#191628` | 低噪声 |
| 选中琥珀 | `#FFD766` | 选房主强调 |
| 资源金币 | `#FFD84D` | 金币 / 收益 |
| 资源电力 | `#57E6FF` | 电力 / 科技 |
| 安全绿色 | `#6FE38B` | 睡觉 / 发育 / 正常门血 |
| 警戒橙色 | `#FF9E45` | 中危 / 接近 |
| 危险红色 | `#FF4D5E` | 破门 / 低门血 |
| 冻结冰青 | `#7AE7FF` | 控制窗口 |
| 紫色功能 | `#A678FF` | 道具 / 特殊效果 |

### 5.2 HUD 组件要求

- 所有文字对深色底对比度要足够，关键数值不得低透明。
- 资源 Chip：图标 + 名称 + 数值，数值优先可读。
- 血条：需要标题、当前比例视觉、危险色变；不能只有颜色无文字。
- 主按钮：每个按钮要有标题、短副标题、图标；禁用态保持可解释。
- 小屏折叠：折叠句柄必须能说明“展开什么”，不能只放箭头。

### 5.3 图形资产规格建议

优先产出以下原创资产，便于快速接入：

| 资产 | 尺寸建议 | 格式 | 透明背景 | 用途 |
|---|---:|---|---|---|
| 宿舍房间地块 4 类 | 512×512 | PNG/WebP | 是 | 选房 / 地图底 |
| 走廊地块 / 暗纹 | 512×1024 | PNG/WebP | 是或否 | 地图背景 |
| 门状态 4 类 | 256×128 | PNG/WebP | 是 | 正常 / 受损 / 危急 / 破门 |
| 床等级 1–5 | 256×192 | PNG/WebP | 是 | 睡觉与升级反馈 |
| 炮台等级 1–5 | 256×256 | PNG/WebP | 是 | 防御单位 |
| 敌人状态 4 类 | 256×256 | PNG/WebP | 是 | 巡查 / 逼近 / 破门 / 狂暴 |
| 冻结敌人覆盖层 | 256×256 | PNG/WebP | 是 | 状态特效 |
| 道具图标 7 类 | 128×128 | PNG/WebP | 是 | speedUp / goldBoost / doorRepair / freezeGhost / invincible / barrier / slowTrap |
| HUD 图标集 | 64×64 / 128×128 | PNG/WebP/SVG-like | 是 | 金币、电力、门、床、炮台、时间、警报 |
| 面板装饰九宫格 | 256×256 | PNG/WebP | 是 | 可选，不强依赖 |

## 6. multi-agent-image 生成内容范围

### 6.1 可以生成的内容

`multi-agent-image` 只用于生成**原创视觉参考与可落地资产草案**，范围如下：

1. **原创 UI moodboard**
   - 选房阶段主界面视觉方向图。
   - 夜间防守阶段主界面视觉方向图。
   - HUD 资源条 / 血条 / 操作按钮组合参考。

2. **原创地图图形资产**
   - 深夜宿舍走廊地块。
   - 4 个差异化宿舍房间地块：安静房、发育房、近门房、高收益房。
   - 门、床、炮台槽位、路径提示、风险标记。

3. **原创单位与状态资产**
   - 玩家睡觉 / 醒来状态的小人或床上轮廓。
   - 敌人原创剪影：巡查、逼近、破门、狂暴、冻结覆盖。
   - 炮台等级视觉、炮击光效、门受击光效。

4. **原创图标资产**
   - 金币、电力、门、床、炮台、时间、警报。
   - 7 类道具图标：加速、金币增益、修门、冻结、无敌、屏障、减速陷阱。

5. **胜负结果视觉参考**
   - 天亮胜利：晨光、宿舍安全、轻庆祝。
   - 失守失败：门破、红黑警戒、无血腥。

### 6.2 生成交付要求

每一批 multi-agent-image 结果必须附带：

- 用途标签：选房 / 夜间防守 / HUD / 图标 / 单位 / 特效。
- 建议接入位置：SwiftUI HUD、SpriteKit 地图节点、Sheet 图标等。
- 透明背景需求：是 / 否。
- 建议裁切区域与安全边距。
- 颜色 Token 对应说明。
- 版权声明：原创生成，不含商业原版素材引用。

### 6.3 禁止生成的内容

严格禁止：

1. **商业原版复刻**
   - 不得生成、临摹或提示模型复刻“猛鬼宿舍”或其他商业游戏的原角色、原怪物、原 UI、原地图、原按钮、原图标、原宣传图。
   - 不得使用“same as / in the style of [具体商业游戏 / 画师 / IP]”之类提示。

2. **受保护 IP 元素**
   - 不得出现知名动漫、游戏、影视角色或其近似造型。
   - 不得生成可识别商标、Logo、品牌名、学校真实徽标。

3. **血腥恐怖过界**
   - 不得生成血腥、肢解、真实恐怖面孔、惊吓图。
   - 敌人应保持卡通化 / 符号化 / 幽影化。

4. **误导玩法的素材**
   - 不得生成玩家在走廊自由移动、追逐、近战攻击等会误导当前玩法的画面。
   - 不得生成复杂开放地图、多个楼层、自由探索 UI。
   - 不得让选房阶段看起来像实时战斗。

5. **不可集成资产**
   - 不接受只有整张概念图、无法拆分的复杂 UI。
   - 不接受文字烘焙在图片里的按钮和 HUD；文字应由 SwiftUI 渲染。
   - 不接受低对比、过暗、缩小后不可读的小图标。

## 8. 验收标准

### 8.1 选房阶段验收

必须全部满足：

- [ ] 玩家打开选房阶段 3 秒内能看出“当前正在选择宿舍”。
- [ ] 已选房间在地图和底部卡片中同时高亮，二者一致。
- [ ] 4 个房间风险差异可通过颜色 / 标记看出，不只依赖房名。
- [ ] 当前选择卡片包含：房名、风险、产出、门耐久。
- [ ] “确认入住并开始夜晚”是页面最明确的下一步 CTA。
- [ ] 明确提示“确认后角色固定 / 不自由走动”。
- [ ] 小屏下底部选房操作区不遮挡核心地图信息超过可接受范围；折叠态仍可恢复。
- [ ] 无任何商业原版素材、相似角色或相似 UI 复刻。

### 8.2 夜间防守阶段验收

必须全部满足：

- [ ] 顶部 HUD 能清晰显示：距天亮、金币、电力、门等级、炮台数、门血、敌人血。
- [ ] 门血是最高优先级防守指标；低血时颜色和提示均明显变化。
- [ ] 敌人状态至少能区分：巡查、逼近、破门、狂暴 / 冻结。
- [ ] 玩家睡觉状态有明确视觉反馈，并与资源增长逻辑一致。
- [ ] 推荐动作同一时间只强调一个主建议。
- [ ] 底部操作按钮在小屏上仍可点，文字不被图形资产干扰。
- [ ] 破门警报明显但不遮挡关键按钮和血条。
- [ ] 炮台、门、床、敌人资产缩小到实际游戏尺寸后仍可识别。
- [ ] 无血腥惊吓图，无商业 IP 借用。

### 8.3 HUD / 色彩验收

必须全部满足：

- [ ] 阶段色统一：选房黄、发育绿、守门橙、危机红、冻结冰青。
- [ ] 关键文字和数字在深色背景上可读，不低透明。
- [ ] 颜色不是唯一信息载体；风险、状态、血量需有形状、文字或图标辅助。
- [ ] SwiftUI 文本不烘焙进图片资产。
- [ ] HUD 组件与 SpriteKit 地图资产风格一致：同一描边厚度、发光强度和圆角语言。

### 8.4 multi-agent-image 资产验收

必须全部满足：

- [ ] 每个文件命名清晰，如 `room_quiet_selected.png`、`enemy_shadow_attacking.png`。
- [ ] 透明背景资产边缘干净，无黑边 / 白边。
- [ ] 资产包内附简短说明：用途、状态、建议尺寸、颜色 Token。
- [ ] 没有真实商标、商业角色、商业游戏 UI 近似复刻。
- [ ] 小尺寸预览通过：64 px 图标、128 px 单位、实际地图尺寸均可辨认。
- [ ] 与当前玩法一致：玩家固定房间，主要操作是睡觉、升级、修门、炮台、防守到天亮。

## 9. 给后续 Agent 的执行提示

1. 若先改代码，优先改 `GameView.swift` 的 HUD 组件色彩 Token 和信息层级，再改 `GameScene.swift` 地图节点视觉。
2. 若先生成图，先产出 moodboard 和资产草案，不要一次性生成大批最终图。
3. 图形资产接入前必须做小尺寸可读性检查。
4. 任何看起来像商业原版的方案直接废弃，不进入实现。
5. 本轮不追求完整动画系统，优先完成静态图形 + 少量状态光效。
